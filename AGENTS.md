# AGENTS.md

A quick-reference for AI agents and humans navigating `go-zenon`. For the full architectural model see [ARCHITECTURE.md](ARCHITECTURE.md).

## Quick Facts

- Reference Go implementation of Zenon Network of Momentum (NoM), Alphanet Phase 0.
- Module path: `github.com/zenon-network/go-zenon`. Go 1.20+.
- Dual-ledger blockchain: per-account chains of *account blocks* anchored by a global chain of *momentums* produced by elected *pillars*.
- Anti-spam = *plasma* (fused QSR or PoW). Protocol upgrades = *sporks*.

## Glossary

Use these terms exactly in godoc comments — no synonyms.

| Term | Definition |
|---|---|
| **Account Block** | An atomic transaction in one account's chain. Send blocks transfer; receive blocks accept. Each transfer is a (send, receive) pair. |
| **Momentum** | A consensus block produced by a pillar, anchoring a batch of account blocks into the global timeline. |
| **Pillar** | A registered validator eligible for election to produce momentums. Weight comes from delegated stake + historical points. |
| **Sentinel** | A non-producing node operator that earns rewards for uptime. |
| **Plasma** | Per-block resource budget that gates account-block creation. Generated by fusing QSR or burning PoW. |
| **Spork** | A protocol-upgrade switch keyed to a momentum-height threshold and committed by the spork address. |
| **Embedded Contract** | A built-in system contract at a fixed address. Each method has `GetPlasma`, `ValidateSendBlock`, `ReceiveBlock`. |
| **VM Context** | The per-block execution environment passed to contract methods; carries chain state, momentum, plasma. |
| **Verifier** | The component that validates account blocks and momentums against consensus rules before insertion. |
| **Tick** | A unit of consensus time; one elected pillar per tick. |
| **Points** | Per-pillar performance score affecting future election weight. |
| **ChainBridge** | The wiring layer connecting `chain` reads/writes to P2P broadcast/sync. |
| **Insert Lock** | The single mutex that serializes all chain mutations (`chain.AcquireInsert`). |
| **Patch** | An atomic, versioned set of state changes from `common/db`. |
| **Token Standard** | Identifier for a ZTS token (analogous to a contract address for an ERC-20). |
| **Coinbase** | The producing key pair configured on a pillar node. |
| **Descendant Block** | A contract-emitted send nested inside the receive block that triggered it. |
| **Mailbox** | Per-account queue of unreceived sends awaiting consumption. |

## Conventions Used Throughout

- **Logging:** structured kv pairs via `common/logs.go`. Each subsystem has its own logger: `common.ChainLogger`, `common.ConsensusLogger`, `common.NodeLogger`, `common.P2PLogger`, `common.VmLogger`.
- **Errors:** common errors in `common/errors.go`; VM-specific in `vm/constants/errors.go`. Wrap with `errors.Wrap` (from `github.com/pkg/errors`) when adding context.
- **Concurrency annotations:** godoc comments use `// Concurrency:` lines to call out goroutine-safety and lock requirements.
- **Invariants:** godoc comments use `// Invariant:` lines for facts that must always hold.
- **Cross-references:** godoc comments use `[pkg.Symbol]` (Go 1.19+ doc-link syntax) liberally.
- **Generated files:** `*.pb.go` files are excluded from manual documentation. They carry the standard `Code generated by protoc; DO NOT EDIT.` marker.
