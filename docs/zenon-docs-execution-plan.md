# go-zenon Developer Docs — Execution Plan

**Goal:** Build a Docusaurus documentation site for go-zenon (https://github.com/zenon-network/go-zenon) where:
1. The Go package reference is generated from godoc comments in the source code (docs live in the code).
2. Hand-written guides/concepts live as MDX, with code snippets injected from real, compile-tested Go files.
3. The JSON-RPC API has an interactive, Swagger-style playground — built on **OpenRPC** (not OpenAPI, because go-zenon's API is JSON-RPC 2.0, not REST).
4. The docs are maximally consumable by AI agents (llms.txt, raw markdown, machine-readable OpenRPC spec, and eventually an MCP server generated from the spec).

**Topology decision (already made):** Separate docs repo (`zenon-docs`) that clones/pins go-zenon at build time. Do NOT require changes to be merged upstream into zenon-network/go-zenon to ship. Where godoc improvements are needed, they go to the fork `0x3639/go-zenon` first and upstream PRs later.

**Reference model:** Sui (MystenLabs/sui) — `docs/content` (MDX) + `docs/site` (Docusaurus app), PR preview deploys, code snippets injected from anchored source files. We adapt that pattern to a standalone repo.

## Phase 0 — Scaffold (do this first, end-to-end deploy of a near-empty site)

**Tasks:**
1. Init Docusaurus (classic preset, TypeScript) in `site/`. Use Docusaurus **v3.x**, pin exact version in package.json. Use pnpm.
2. Configure `docs` plugin to read from `../content` (set `path: '../content'`, `routeBasePath: 'docs'`).
3. Create `PINNED_GO_ZENON_REF` containing the latest go-zenon release tag.
4. Write `scripts/fetch-go-zenon.sh`: shallow-clone `https://github.com/zenon-network/go-zenon` at the pinned ref into `.build/go-zenon` (gitignored). Idempotent.
5. Set up GitHub Actions:
   - `build-deploy.yml`: on push to main → fetch go-zenon → run generators (later phases) → `pnpm build` → deploy to Cloudflare Pages (or GitHub Pages — pick Cloudflare Pages if a CF account token is available, it gives PR previews for free).
   - `preview.yml`: on PR → same build → preview URL posted as PR comment.
6. Stub `content/intro/index.md` so the site builds.

**Acceptance:** Site builds locally with `pnpm build` and deploys green from CI.

## Phase 1 — godoc → package reference (gomarkdoc)

**Tasks:**
1. `scripts/gen-godoc.sh`:
   - Requires Go toolchain (match go-zenon's go.mod version).
   - Install `github.com/princjef/gomarkdoc/cmd/gomarkdoc@latest` (pin a version).
   - Run gomarkdoc over the important packages in `.build/go-zenon` — at minimum: `chain`, `consensus`, `verifier`, `vm`, `vm/embedded/...`, `protocol`, `rpc/api/...`, `wallet`, `zenon`, `common/types`. Walk `./...` and skip `internal/` and test-only packages.
   - Output one `.md` per package into `content/reference/packages/<import-path>.md`.
   - Use a gomarkdoc template/post-process step to add Docusaurus frontmatter (`title`, `slug`, `sidebar_label`) and to escape MDX-hostile characters (gomarkdoc output can contain `{`, `<` sequences that MDX v3 chokes on — sanitize: wrap problem segments in backticks or escape them; this is the most common failure mode, handle it deliberately).
4. Add `content/reference/packages/` to `.gitignore`; generation happens in CI and in a local `pnpm run gen` task.
5. Auto-generate the sidebar for the packages section (Docusaurus autogenerated sidebar over the folder is fine; preserve package path hierarchy).
6. Each generated page must include a "View source" link to the file on GitHub at the pinned ref.

**Acceptance:** `pnpm run gen && pnpm build` produces a browsable package reference; no MDX compile errors; spot-check 3 packages render correctly (`vm/embedded`, `rpc/api`, `chain`).

## Phase 2 — Guides + compile-tested snippet injection (Sui pattern)

**Tasks:**
1. Create `examples/` as a Go module that imports go-zenon at the pinned version. Every code snippet shown in guides must come from this module.
2. Anchor convention in Go files:
   ```go
   // docs::start:connect-client
   client, err := rpc.NewClient("ws://127.0.0.1:35998")
   // docs::end:connect-client
   ```
3. Write a small remark plugin (in `site/src/plugins/inject-snippets/`) that resolves
   `{@inject: examples/path/file.go#connect-client}` directives in MDX at build time:
   - Reads the file, extracts between the start/end anchors, strips the anchor lines, dedents.
   - Renders as a fenced ```go block with a `title` attribute linking to the source file on GitHub.
   - **Build fails hard** if the file or anchor is missing.
   (Sui has an equivalent plugin in MystenLabs/sui under `docs/site` — use it as a reference implementation, but a from-scratch ~150-line remark plugin is acceptable and avoids the dependency.)
4. CI step in `validate.yml`: `cd examples && go build ./... && go vet ./...` (and `go test ./...` where examples are testable). Stale or broken snippets = red build.
5. Author the initial guide set (skeletons with real injected snippets where possible):
   - `tutorials/run-a-node.mdx`
   - `tutorials/first-transaction.mdx` (send ZNN, receive blocks — explain the receive step, it surprises everyone coming from account-balance chains)
   - `guides/fuse-plasma.mdx`, `guides/generate-pow.mdx`
   - `guides/run-a-pillar.mdx`, `guides/run-a-sentinel.mdx`
   - `concepts/dual-ledger.mdx`, `concepts/momentums.mdx`, `concepts/plasma.mdx`, `concepts/embedded-contracts.mdx`

**Acceptance:** At least 3 guides ship with injected snippets; deleting an anchor in `examples/` breaks the build; `go build ./examples/...` runs in CI.

## Phase 3 — OpenRPC spec + interactive playground (the Swagger equivalent)

### 3a. Author the spec

1. YAML fragments in `specs/openrpc/`, one file per RPC namespace. Schema components (Address, Hash, HashHeight, AccountBlock, Momentum, AccountInfo, TokenStandard, paginated list wrappers) live in `specs/openrpc/components/` and are `$ref`'d.
2. Source of truth for method signatures: `.build/go-zenon/rpc/api/` (and `rpc/api/embedded/`). Read the Go handler structs and their godoc comments; every public method becomes one OpenRPC `method` entry with:
   - `name` (e.g. `ledger.getFrontierMomentum`, `embedded.pillar.getAll`)
   - `params` with JSON schemas and descriptions
   - `result` schema
   - at least one `examples` pair with a **real captured response** (run a local/public node and capture actual output — do not invent example responses; agents and humans both pattern-match on these)
   - pagination params documented consistently (`pageIndex`, `pageSize`, max page size).
3. `scripts/assemble-openrpc.mjs`: merge fragments → single `specs/openrpc.json` (OpenRPC 1.2.x, `info.version` = go-zenon pinned tag), validate with `@open-rpc/schema-utils-js`. Commit the assembled JSON. Copy into `site/static/openrpc.json` at build.
4. **Priority order for authoring** (don't try to do all ~14 namespaces in one pass): `ledger` → `embedded.pillar` → `embedded.plasma` → `embedded.token` → `embedded.stake` → `embedded.sentinel` → `stats` → remaining embedded namespaces → `subscribe` (WebSocket subscriptions: document as methods but mark `x-transport: websocket`; the playground will be HTTP-only initially).

### 3b. Render + playground

1. Integrate `@metamask/docusaurus-openrpc` pointed at `specs/openrpc.json`, mounted under `reference/json-rpc`. **Known risks — handle explicitly:** it is a low-download-count package with reported Docusaurus v3 compatibility issues and a hard requirement that at least one markdown file exists in the target reference directory. Pin its version. Timebox integration to one focused attempt.
2. **Fallback (build this if 3b.1 fights back):** custom `RpcPlayground` MDX component + per-method pages generated from `openrpc.json` by a small script (method pages are just MDX with frontmatter; the component renders a param form from the JSON schema, builds the JSON-RPC 2.0 envelope, POSTs via fetch, pretty-prints response). This is ~1–2 days of work and removes the dependency risk entirely. `@open-rpc/docs-react` components may be reused piecemeal.
3. Playground node selector (persisted in localStorage): `Public node (default)` / `Custom URL` / `localhost:35997`. Every method page also shows a copy-pasteable `curl` example of the raw JSON-RPC envelope.

### 3c. Conformance testing

1. `scripts/conformance-test.mjs`: for every method in the spec with an example, POST the example request to a node URL from env (`ZENON_NODE_URL`), validate the response against the result schema (ajv). Allowlist methods that can't run statelessly. Run in `validate.yml` weekly + on spec changes (against the public read-only node).

### 3d. Infrastructure note (manual step, flag for the operator — not for Cursor to execute)

The playground needs a read-only public node with CORS allowing the docs origin: go-zenon config `RPC.HTTPCors: ["https://<docs-domain>"]` (or `*`), fronted by Caddy with rate limiting — same pattern as test.hc1node.com. Document the required config in `README.md` and leave `ZENON_NODE_URL` as a repo variable.

**Acceptance:** Spec validates; ≥3 namespaces fully authored with real examples; a user can open a method page, edit params, hit a node, and see the live response; conformance test passes against the public node.

## Phase 4 — AI consumability

1. Add `docusaurus-plugin-llms` (or equivalent) → generate `llms.txt` and `llms-full.txt` at the site root. `llms.txt` must link to: the OpenRPC spec URL, the package reference index, and each guide.
2. Expose raw markdown: ensure every doc page is reachable as plain `.md` (plugin or build step that copies sources/rendered markdown into `static/raw/`). Add the copy-page-as-markdown button (`docusaurus-plugin-copy-page-button` — same one used by Ethereum and Sui docs).
3. Stable URLs: never rename slugs without `@docusaurus/plugin-client-redirects` entries.
4. Add an `AGENTS.md` at the repo root of zenon-docs describing: build commands, generation pipeline, where things live, and the rule that reference packages are generated (never hand-edit).
5. **Stretch (separate work item, design only in this repo):** generate an MCP server from `openrpc.json` — one MCP tool per RPC method, input schemas mapped from the spec, pointed at the read-only node. Write a short design doc at `content/contribute/mcp-server.md`; implementation lives in its own repo later.

**Acceptance:** `curl https://<site>/llms.txt` and `/openrpc.json` both return useful machine-readable artifacts; copy-as-markdown works on guide and reference pages.

## Phase 5 — Versioning + polish

1. Wire `PINNED_GO_ZENON_REF` into Docusaurus versioning: when go-zenon tags a release, bump the ref, regenerate, and cut a Docusaurus version snapshot. Keep only `current` + latest release to limit build size.
2. Search: Algolia DocSearch (apply — OSS program) or `@easyops-cn/docusaurus-search-local` as the zero-dependency default. Start with local search.
3. Link checking in CI (`docusaurus build` warnings as errors: `onBrokenLinks: 'throw'`).

## Execution order for Cursor

Work strictly in this order; each phase must build green before starting the next:

1. **Phase 0** — scaffold + CI deploy (half day)
2. **Phase 1** — gomarkdoc pipeline (1 day; budget time for MDX escaping issues)
3. **Phase 3a for `ledger` + `embedded.pillar` only** — prove the spec format early (1–2 days)
4. **Phase 3b** — playground integration, with the fallback decision made inside a timebox (1–2 days)
5. **Phase 2** — examples module + snippet injection + first guides (2–3 days)
6. **Phase 3a remainder + 3c** — fill out namespaces + conformance tests (ongoing)
7. **Phase 4** — AI consumability (half day)
8. **Phase 5** — versioning + search (half day)

## Global constraints

- Pin every tool version (gomarkdoc, Docusaurus, the OpenRPC plugin, Node, Go, pnpm). No floating `@latest` in CI.
- Generated content is never hand-edited and never committed (except the assembled `openrpc.json`, which IS committed because it's a published artifact).
- Every build failure mode should be loud: missing anchors, invalid spec, broken links, MDX errors all fail CI.
- Do not invent RPC example responses — capture them from a real node.
- All snippets shown in docs must come from the `examples/` Go module that compiles in CI.
